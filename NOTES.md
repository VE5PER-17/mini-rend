## 工具链 / 构建相关

### vcpkg(包管理器)
- **是什么:** C++ 的包管理器,类似 Java 的 Maven。告诉它"装 GLFW",它自动下载、编译、配好路径。
- **关键点:** 装在 `C:\dev\vcpkg`(全局共享,不放项目里)。和 CMake 配合靠 **toolchain 文件**:`C:\dev\vcpkg\scripts\buildsystems\vcpkg.cmake`。CMake 用了这个 toolchain,`find_package` 才能找到 vcpkg 装的库。
- **踩过的坑:** 无。

### CMake(构建系统)
- **是什么:** C++ 的构建系统,读 `CMakeLists.txt` 生成具体的编译指令。
- **关键点:** CMake **不自动扫描源文件**(故意的设计,C++ 生态太复杂,自动扫易错)。要手动 `add_executable(name src/a.cpp src/b.cpp)` 列出源文件。`file(GLOB)` 能自动收集但**不推荐**——加文件不生效、删文件报幽灵错。
- **踩过的坑:** 无。

### compile_commands.json
- **是什么:** CMake 导出的文件,记录每个 .cpp 编译时的完整命令(含所有 include 路径)。
- **关键点:** 解决 VS Code 红线的关键。CMakeLists.txt 开 `CMAKE_EXPORT_COMPILE_COMMANDS ON` 生成它,`.vscode/c_cpp_properties.json` 里设 `"compileCommands"` 指向它。比手动配 include 路径更准,且自动跟随 CMake 变化。
- **踩过的坑:** 无。

## C++ 语言相关(Java→C++ 迁移)

### 指针 vs 引用
- **是什么:** 指针是"存地址的变量",引用是"别名"。Java 只有引用(对象都在堆),C++ 两者都有,还有值。
- **踩过的坑:** `GLFWwindow*` 是**不透明句柄(opaque handle)**,类似存衣处号码牌——不是 buffer 指针,你也看不到它指向的对象内部。绝不解引用(`*window`)。Vulkan 里全是这种 handle(`VkInstance` 等)。

### 两个指针经典坑(必须分清)
| 坑 | 是什么 | 危害 | 防法 |
|---|---|---|---|
| **未初始化垃圾值** | 指针没初始化,不是 null 是随机 bit | 当成有效地址用 → 访问随机内存崩溃 | 类内默认初始化 `= nullptr` |
| **垂悬指针(dangling)** | 指针指向的内存已释放,指针还留着旧地址 | 读写已释放内存 → 崩溃或静默 corrupt | 析构后置 `= nullptr` |

- **Java 类比:** Java 引用默认 null,不会有"垃圾值";释放后引用还在但 GC 不会再让你访问(C++ 没这保护)。
- **记住:** C++ 没初始化的指针不是 null,是垃圾。永远显式初始化。

### RAII(资源获取即初始化)
- **是什么:** 把资源绑到对象生命周期——构造函数获取,析构函数释放。对象一死资源自动放。
- **关键点:**
  - 析构**必定**调用(栈展开 stack unwinding):正常 return、提前 return、甚至抛异常,已构造的对象都会被析构。这是 C++ 语言规则强制的,所以你"想忘都忘不掉"——这是 RAII 可靠的根本。
  - 析构顺序 = 构造顺序的**逆序**(栈 LIFO/FILO)。后构造的先析构。这个顺序经常恰好是对的(如先关窗再终止库)。
  - **栈对象优先,智能指针按需。** 能用栈就别堆。`unique_ptr` 留给"需要堆/共享/多态"的场景。强行用智能指针是 Java 残留(Java 对象默认堆)。
- **Java 类比:** 像 `try-with-resources`,但不需要 try 块包着,且时机精确到作用域结束瞬间(不像 GC 时机不定)。
- **怎么写:** 构造获取、析构释放、禁用拷贝、指针成员 `= nullptr` 初始化、析构后置 nullptr。

### 三/五法则(管理资源的类要管好拷贝)
- **是什么:** 类若管理资源(指针/句柄/文件),必须决定拷贝行为:要么正确实现深拷贝,要么禁掉。
- **关键点:** Window 类禁用拷贝——否则两个对象指向同一 `GLFWwindow*`,析构时双重销毁崩溃。写法:`Window(const Window&) = delete; Window& operator=(const Window&) = delete;`

### 作用域与栈对象生命周期
- **是什么:** 栈对象在离开所在 `{}` 作用域时析构。
- **关键点:** 这就是 RAII 的触发机制。

### 头文件组织
- **是什么:** 声明放 .h,实现放 .cpp。`#pragma once` 防重复包含。
- **关键点:** 短函数可直接写在类定义里(inline)。`Window::Window` 的 `::` 是作用域解析,表示"这是 Window 类的函数"。成员变量下划线后缀(`window_`)区分局部变量。const 成员函数(后缀 const)表示不改对象。

## 图形渲染相关

### 渲染管线(rendering pipeline)
- **是什么:** GPU 把顶点数据变成屏幕像素的流水线。输入顶点,过几个工位,输出像素。
- **关键工位:**
  1. 顶点着色器(可编程)— 逐顶点,把坐标变换到屏幕坐标
  2. 图元装配(固定)— 每3顶点拼三角形
  3. 光栅化(固定)— 三角形涂成像素
  4. 片段着色器(可编程)— 逐像素,决定颜色
  5. 输出
- **关键认知:**
  - 着色器用 **GLSL** 写(GPU 专用语言,语法像 C,有 vec3/vec4 等类型)。
  - 着色器**对每个元素并行跑**(顶点着色器逐顶点,片段着色器逐像素)。GPU 为大规模并行设计,这是它快的原因。
- **待补原理:** 坐标系变换(模型→世界→视图→裁剪→屏幕,NDC)、齐次坐标(vec4)、光栅化规则。
